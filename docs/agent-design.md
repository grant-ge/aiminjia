# AI小家 — 组织专家，工作助手 Agent 设计文档

## 一、产品概述

**产品名称**：AI小家
**一句话定位**：面向 HR 的 AI 薪酬分析 Agent，通过自然对话完成薪酬公平性分析和日常 HR 咨询。

**核心能力**：

- **日常咨询**：新人定薪、晋升调薪、竞对 offer 应对、预算分配、政策咨询
- **深度分析**：上传工资表后自动进入 5 步分析流程（数据清洗 → 岗位归一化 → 职级推断 → 公平性诊断 → 行动方案），每步人机确认
- **文档解析**：支持 Excel、Word、PDF、CSV 文件上传，自动解析内容
- **代码执行**：内置 Python 运行时，自动写代码处理大文件和复杂数据
- **联网搜索**：遇到知识盲区自动搜索行业薪酬、法规政策等信息
- **文档生成**：在工作目录中生成 HTML/PDF/DOCX/Excel 报告

**产品形态**：
- 桌面端应用（Mac + Windows），Tauri 2.x
- 对话式交互，支持文件上传
- 内置 Python 运行时
- 本地工作目录读写
- 多模型支持：DeepSeek（默认）、火山引擎、OpenAI、Claude、Qwen、自定义模型（OpenAI-compatible）

### 2A.3 自然语言定制能力

用户可以通过自然语言对预置模板进行调整，Agent 解析意图后实时生效：

**岗位族定制示例**：
```
用户："把品质合并到生产里，一共7个族"
→ Agent：合并 Q-品质 到 M-生产，更新为 M-生产品质

用户："我们有一个独立的海外事业部，需要单独一个族"
→ Agent：新增 OV-海外业务 岗位族
```

**职级通道定制示例**：
```
用户："四序列，但销售和客服要分开，所以是5个序列"
→ Agent：从四序列模板派生，拆分 S 为 S-销售 和 CS-客服

用户："操作序列不需要那么多级，2级就够了"

V1 内置 **DeepSeek** 作为默认推理引擎，能力覆盖：

### 2B.2 搜索增强

```
搜索增强触发场景：
  1. 行业薪酬基准不确定 → 搜索行业薪酬报告

  3. 特殊岗位市场定价 → 搜索招聘网站薪酬范围

  - 自动判断是否需要搜索（置信度低于阈值时触发）

  - 结果标注来源，保持透明
  - 缓存机制：同类问题 7 天内不重复搜索

## 二-C、文件处理与代码执行能力

### 2C.1 文件上传与解析

```
支持的文件类型：

  Excel（.xlsx, .xls）：
    - 小文件（<5MB / <10万行）→ 直接用 openpyxl/pandas 读取
    - 大文件（>5MB 或 >10万行）→ Agent 写 Python 代码分片处理
    - 工资表识别：检测到薪酬相关字段时，主动提议做深度分析

  Word（.docx）：提取全文内容 + 表格结构

  PDF（.pdf）：
    - 文本型 PDF：直接提取文字
    - 扫描型 PDF：提示用户内容可能不完整

  CSV（.csv）：pandas 读取，自动检测编码和分隔符

文件处理流程：
  1. 用户上传 → 保存到工作目录的 uploads/
  2. 自动检测文件类型和大小
  3. 小文件 → 直接解析，提取摘要展示
  4. 大文件 → Agent 写 Python 代码处理
  5. 解析结果纳入对话上下文
  6. 支持同一对话中上传多个文件
```

### 2C.2 代码执行能力

```
自动触发代码执行的场景：
  1. 大文件处理
  2. 复杂数据清洗
  3. 统计分析（回归、聚类、显著性检验）
  4. 数据可视化
  5. 报告生成（Excel/HTML 文件）
  6. 批量处理多文件汇总

代码执行设计：
  - 代码在对话中可见（用户可审查）
  - 执行结果直接展示在对话中
  - 生成的文件保存到工作目录
  - 执行失败时自动修复代码并重试
```

### 2C.3 工作目录管理

```
目录结构（自动创建）：
  工作目录/
  ├── uploads/          # 用户上传的原始文件
  ├── reports/          # 生成的最终报告
  ├── charts/           # 生成的图表
  ├── exports/          # 导出数据
  └── temp/             # 临时文件

权限范围：
  - Agent 仅在用户指定的工作目录内读写
  - 所有文件操作在对话中告知用户
```

   ┌─────────┬──────┬──────┬──────┐

## 四、Step 1：数据清洗与理解

### 4.1 自动数据摄入

Agent 接收用户上传的 Excel 工资表后，自动完成：

1. **字段语义识别**：基于表头名称 + 数据特征，映射到标准语义
   ```
   "基本工资"   → base_salary（固定薪酬-基础）
   "绩效基数"   → perf_base（固定薪酬-绩效基准）
   "绩效系数"   → perf_coefficient（浮动因子）
   "保密工资"   → confidential_pay（固定薪酬-补贴）
   "销售提成"   → commission（浮动薪酬-提成）
   "工时加班工资" → overtime_pay（浮动薪酬-加班）
   ```

2. **薪酬结构拆解**：自动将薪酬分为三层
   - **固定薪酬** = 基本工资 + 保密工资 + 确认为固定的津贴
   - **浮动薪酬** = 绩效工资(基数x系数) + 加班费 + 提成
   - **一次性收入** = 奖金、现金补贴

3. **薪酬结构公式检测**：自动检测企业是否有隐性的固定比例

4. **身份兼岗检测**：识别一人多职位、管理+专业双重角色

### 4.2 数据清洗规则

以下人员默认排除出分析样本（向用户展示排除原因，可逐项覆盖）：

| 排除规则 | 判断条件 | 原因 |
|---------|---------|------|
| 当月入职（未满整月） | 入职日期在账期月份内 | 非完整月薪，扭曲统计 |
| 当月离职 | 状态=已离职 | 同上 |
| 非全职用工 | 用工类型 ∈ {实习, 劳务派遣, 退休返聘} | 薪酬定价逻辑不同 |
| 长期请假 | 请假时长 > 应出勤天数 x 30% | 实发工资被大幅压低 |
| 试用期 | 入职日期 ≤ 3个月 且 无转正日期 | 薪资可能打折，打标签保留 |
| 极端异常 | 基本工资 = 0 | 数据质量问题 |

### 4.3 数据质量报告

向用户展示完整的数据质量评估，包括覆盖率、缺失字段、以及缺失字段对分析的影响。

2. 排除清单 → "以下人员建议排除，是否同意？"
3. 数据质量报告 → "是否有补充数据可以提供？"

## 五、Step 2：岗位归一化与岗位族构建

### 5.1 岗位族——推断 → 匹配模板 → 客户定制

**Step A：行业特征推断**

Agent 从数据中自动推断企业行业特征：
```
推断信号：
  - 部门名称关键词：车间/生产/装配 → 制造业特征
  - 职位名称关键词：外贸/国际/海外 → 外贸型企业
  - 薪酬结构特征：大比例提成 → 销售驱动型
  - 人员分布：操作工占比 > 30% → 劳动密集型
  - 总人数 → 规模判断
```

**Step B：基于模板库推荐方案**

Agent 根据推断的行业+规模，从预置模板库中自动匹配 2~3 个方案，按匹配度排序展示。

**Step C：自然语言定制**

用户可以在任何预置方案基础上，通过自然语言自由调整。

**Step D：LLM 语义聚类 + 薪酬验证**

在客户选定岗位族方案后：
1. LLM 对全部原始职位名称做语义分组，归入客户选定的岗位族
2. 识别地域/业务线前缀（作为维度标签，不作为岗位区分）
3. 用薪酬分布重叠度验证分组合理性
4. 对置信度低的归类，标记给用户确认

### 5.2 岗位归一化

核心规则：

1. **去地域/业务线前缀**：`国际客服工程师` → `现场客服工程师`，保留原始前缀作为维度标签
2. **合并同义职位**：`机加工` = `机加工钳工` = `机加工车床`（需用户确认）
3. **保留层级差异**：`客服工程师` 和 `区域客服经理` 不合并
4. **保留专业差异**：`机械工程师` 和 `电气工程师` 不合并
5. **识别双重角色**：管理兼专业岗位标记处理

### 确认卡点 2

按岗位族逐批向用户展示归一化映射，请求确认。支持点击编辑修改标准岗位名称或岗位族归属。

## 六、Step 3：职级推断与定级

### 6.1 三阶段推断法

```
阶段 A：非薪酬信号 → 粗职级
    ↓
阶段 B：薪酬聚类 → 细分子级
    ↓
阶段 C：交叉验证 → 标记"职级-薪酬不一致"的异常个体
```

### 6.2 阶段 A：职级通道框架

Agent 根据已推断的行业特征，从预置模板库中推荐最匹配的通道方案，用户可通过自然语言调整。

### 6.3 阶段 A（续）：基于非薪酬信号的粗职级推断

参考美世 IPE 简化模型，从已有数据中提取可推断的因素：

| IPE 因素 | Agent 可用信号 | 推断方法 |
|---------|-------------|---------|
| **影响** | 管理关键词 + 部门规模 + 汇报层级 | LLM 语义判断 + 部门人数统计 |
| **沟通** | 岗位族属性 | 按岗位族设定默认值 |
| **创新** | 专业关键词 + 岗位族 | 工程师/设计 > 专员 > 操作工 |
| **知识** | 管理幅度 + 专业复杂度 | 综合推断 |

Agent 的推断逻辑基于通用信号映射到客户配置的级别框架中，非硬编码。

### 6.4 阶段 B：基于薪酬聚类的子级细分

对同一 `标准岗位 x 粗职级` 组合内的员工，按固定薪酬做自然断点分析。

### 6.5 阶段 C：交叉验证与异常标记

用司龄作为独立验证维度，标记个体异常。

### 6.6 地域差异处理

```
Agent 自动检测工作地点维度：
  - 如有"工作地点"字段 → 按城市分组对比中位薪酬
  - 如有不同城市的同岗位员工 → 计算地域差异系数
  - 在诊断阶段：控制地域因素后再判断公平性
```

### 确认卡点 3

逐岗位族展示定级结果和异常标记，支持逐个确认或批量确认。

## 七、Step 4：薪酬公平性诊断（含根因分析）

### 7.1 分析框架

**V1 可控制的因素**：

| 控制因素 | 数据来源 | 说明 |
|---------|---------|------|
| 标准岗位 | Step 2 产出 | 岗位价值差异 |
| 推断职级 | Step 3 产出 | 能力/贡献层级差异 |
| 岗位族 | Step 2 产出 | 职能领域差异 |
| 司龄 | 入职日期计算 | 经验积累 |
| 工作地点 | 原始数据 | 地域薪酬差异 |
| 部门 | 原始数据 | 组织内差异 |

**V1 无法控制的因素**：
- 性别/年龄、学历/资质证书、绩效评级历史

### 7.2 六维度分析

#### 维度 1：岗位内部公平性（同岗同酬）
方法：对每个 `标准岗位 x 职级` 组合计算 CV、极差比、四分位距比。

#### 维度 2：跨岗位公平性
方法：同一职级内，不同标准岗位的中位固定薪酬对比。

#### 维度 3：薪酬-司龄回归分析
方法：参考美世回归分析方法论，采用 1.65 SD 标准。

#### 维度 4：薪酬倒挂检测
方法：在同 `标准岗位 x 职级` 组内，对比不同入职年份群组的中位薪酬。

#### 维度 5：薪酬结构合理性
方法：检查固定/浮动比例是否与岗位性质匹配。

#### 维度 6：内部 Compa-Ratio 分析
方法：内部 Compa-Ratio = 员工固定薪酬 / 同组中位薪酬 x 100%。

### 7.3 根因分析

对每个异常，Agent 自动进行根因推断：

```
根因分析框架：

高优先级异常：23 人显著偏低

  根因分析结果：
  ├─ 12 人 → 入职定薪偏低 + 后续无调薪机制
  │   特征：司龄 3-8 年，入职时市场水平低
  │   根因：企业缺乏年度调薪机制
  │
  ├─ 6 人 → 岗位职责已升级但薪酬未跟上
  │   根因：缺乏定期岗位评估机制
  │
  ├─ 3 人 → 地域差异未在定薪中体现
  │   根因：缺乏地域薪酬差异政策
  │
  └─ 2 人 → 特殊情况需人工判断

高优先级：8 人严重薪酬倒挂

  根因分析结果：
  ├─ 5 人 → 近两年市场溢价招聘导致
  │   根因：外部招聘按市场行情定价，内部存量员工无对应调薪
  │
  └─ 3 人 → 跨部门调动后薪酬未调整
```

### 7.4 诊断结果汇总

```
薪酬公平性诊断总览

整体健康指标
  全员固定薪酬 Gini 系数: 0.32（中等不平等）
  职级-薪酬相关性 R²: 0.75
  薪酬在合理区间: 84.6%（873 / 1,032 人）
  薪酬倒挂率: 12.3%（127 例）

根因诊断摘要
  1. 缺乏系统性年度调薪机制 → 导致 60% 的严重偏低问题
  2. 外部招聘市场定价 vs 内部存量未同步 → 导致 80% 的薪酬倒挂
  3. 岗位职责升级未同步调级调薪 → 导致 25% 的偏低问题
  4. 缺乏地域差异薪酬政策

建议的制度建设
  1. 年度调薪制度（预算建议 3-5% 薪酬总额）
  2. 新人入职公平性触发检查机制
  3. 薪酬带制度（每个职级x岗位有明确的 Min-P50-Max）
  4. 地域薪酬系数表
  5. 年度岗位评估流程
```

### 确认卡点 4

展示诊断结果 + 根因分析，逐级确认：
1. 整体健康度评估
2. 高优先级异常 + 根因分析
3. 制度建设建议

## 九、数据安全与合规

### 9.1 安全设计原则

```
1. 本地优先：所有数据存储在用户本地，不上传到任何云端
2. LLM 调用脱敏：发送给 LLM API 的数据经过脱敏处理
3. 审计日志：所有操作留痕，可追溯
4. 最小权限：Agent 只访问分析所需的最小数据集
```

### 9.2 LLM 数据脱敏策略

```
脱敏覆盖范围：
  1. 用户消息：gateway 层脱敏后发送给 LLM
  2. 工具执行结果：Tool 返回的文件数据经脱敏后回传 LLM
  3. 最终响应还原：LLM 返回的内容保存前通过 MaskingContext 反向还原

脱敏字段：
  - 员工姓名 → 替换为工号（#9262）
  - 部门名称 → 可选脱敏
  - 公司名称 → 可选脱敏
  - 手机号码 → 严格模式下替换为占位符
  - 邮箱地址 → 严格模式下替换为占位符

保留字段（分析必需）：
  - 薪酬数值
  - 岗位名称
  - 入职日期/司龄
  - 工作地点

脱敏级别：严格 / 标准 / 宽松
```

### 9.3 审计日志

```
记录内容：
  - 数据上传时间和文件名
  - 每步分析的操作记录
  - 用户确认/修改的决策记录
  - LLM API 调用记录
  - 报告导出记录

存储位置：本地
保留期限：默认 12 个月
```

### 9.4 PIPL 合规

```
  - 数据采集最小化
  - 明确告知数据处理方式
  - 本地存储，不上传云端
  - 用户可随时删除所有本地数据
  - 发送给 LLM 的数据经过脱敏
```
